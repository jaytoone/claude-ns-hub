#### 1. 다양한 commit 스타일에서 "중요한 결정 커밋" 자동 인식

현재 연구 기반으로 3-tier 접근이 가장 실용적이다:

**Tier 1 — 규칙 기반 (deterministic, zero-cost)**

Conventional Commits 타입 존재 시: `feat`, `fix`, `refactor`, `perf`는 decision signal이 높음. `chore`, `style`, `ci`는 noise 확률 높음. 단, ICSE 2025 연구에 따르면 `chore`가 가장 분류 어렵고 실제로는 significant change를 담기도 함 [FACT-1].

비-CC 저장소에서: Linux 커널 데이터 기준 decision-signal 동사 목록 사용 [FACT-2]:

```python
DECISION_VERBS = {"add", "use", "remove", "introduce", "replace", "switch",
                  "migrate", "refactor", "drop", "implement", "redesign"}
NOISE_INDICATORS = {"fix typo", "bump version", "merge branch",
                    "update changelog", "revert", "wip"}
```

**Tier 2 — 구조적 시그널 (Lore Protocol 기반)**

git trailer 존재 여부로 decision commit 식별 [FACT-3]:

```bash
git log --format="%H %s" | xargs -I{} git show -s --format="%B" {} | \
  grep -E "^(Constraint|Rejected|Directive|Scope-risk):" 
```

**Tier 3 — LLM 분류 (고비용, 고정밀)**

ICSE 2025 결과: CodeLlama fine-tuned (message + diff 동시 입력) macro F1=0.7641 [FACT-1]. "decision vs noise" 이진 분류는 10-타입 분류보다 쉬우므로 실제로는 더 높은 F1 기대 가능.

#### 3. BM25의 open-set recall 저하 이유와 개선 방법

**근본 원인 3가지:**

1. **Vocabulary mismatch** [FACT-9]: 쿼리와 문서가 다른 용어 사용 시 완전 miss. git commit의 경우 저장소마다 도메인 특수어(함수명, 라이브러리명)가 다름 → open-set에서 IDF가 새 저장소에 맞지 않음.

2. **IDF corpus 불일치** [FACT-1, CTX CLAUDE.md]: BM25Okapi는 학습 코퍼스 IDF에 의존. 소규모 새 저장소(commit 수 100-500개)에서 도메인 핵심 용어가 multiple docs에 출현 → IDF 낮아짐 → 정작 중요한 용어가 하향 랭크.

3. **SPLADE의 out-of-domain 역효과** [FACT-10]: 학습된 sparse model을 미지 저장소에 적용 시 BM25보다 나쁠 수 있음.

**개선 방법 (효과 순):**

```
1. Per-repository BM25 재학습
   - 새 저장소 clone 시 해당 저장소 commits만으로 BM25 index 재생성
   - CTX가 이미 이 방식 사용 — 문제는 소규모 코퍼스에서 IDF 왜곡

2. TF-only BM25 (IDF 없는 버전) — 소규모 코퍼스 특화
   - rank_bm25의 BM25L 또는 직접 TF-only 구현
   - CTX CLAUDE.md: keyword R@3 0.379 → 0.724 (IDF 제거 후)

3. Hybrid BM25 + dense embedding [FACT-11]
   - 각 top-20 취합 후 RRF (Reciprocal Rank Fusion)
   - 기술 문서 가중치: BM25 0.6 / dense 0.4
   - 새 저장소: BM25 비중 높임 (embedding OOV 더 심각)

4. Subword-level BM25 [FACT-13]
   - tiktoken으로 tokenize → camelCase/snake_case 함수명 분해
   - "getUserById" → ["get", "user", "by", "id"] matching 가능

5. Query/Doc expansion (Doc2Query)
   - commit message → plausible queries 생성 후 index에 추가
   - 고비용이지만 recall 개선 효과 큼
```

#### 4. Commit message 품질 자동 분류 (decision vs noise)

연구 기반 scoring 시스템:

**Signal-positive 지표:**
- Decision-type 동사 존재: `add`, `introduce`, `replace`, `migrate`, `switch` [FACT-2]
- Lore trailers 존재: `Constraint:`, `Rejected:`, `Directive:` [FACT-3]
- commit body 존재 (subject only vs. subject + body)
- CC 타입이 `feat`/`refactor`/`perf`이고 scope 명시됨 [FACT-1]
- 문장 수 ≥ 2 (rationale density 연구상 ~60%는 >= 2문장) [FACT-2]

**Noise 지표:**
- "merge", "revert", "bump", "wip", "typo" 키워드
- 단일 단어 or 10자 미만 subject
- CC 타입 `style`, `ci`, `chore` + scope 없음
- "[skip ci]" 또는 bot 커밋 (dependabot, renovate)

**실용적 scoring 예시:**

```python
def score_commit_decision(message: str, diff_stat: dict) -> float:
    score = 0.0
    msg_lower = message.lower()
    
    # Lore trailers (강한 signal)
    if any(t in message for t in ["Constraint:", "Rejected:", "Directive:"]):
        score += 0.4
    
    # Decision verbs in subject line
    DECISION_VERBS = {"introduce", "migrate", "replace", "redesign", "implement",
                      "switch", "refactor", "add", "remove"}
    subject = message.split("\n")[0].lower()
    if any(v in subject for v in DECISION_VERBS):
        score += 0.2
    
    # Has body (not just subject)
    if "\n\n" in message and len(message.split("\n\n")[1].strip()) > 20:
        score += 0.2
    
    # Noise signals
    NOISE = {"merge branch", "bump", "wip:", "typo", "revert", "[skip ci]"}
    if any(n in msg_lower for n in NOISE):
        score -= 0.5
    
    # Bot commits
    if any(bot in msg_lower for bot in ["dependabot", "renovate", "github-actions"]):
        score = 0.0
    
    return max(0.0, min(1.0, score))
```

### Latest Trends (2025-2026)

1. **Lore Protocol (2026-03)**: git trailers를 AI agent용 structured decision storage로 사용하는 표준화 시도. 실증 검증 미완이나 방향성은 유효 [S1].

2. **Copilot Memory GA (2026-03)**: citation-verified, 저장소 단위 지식 persistence가 상용 수준으로 구현됨. Cross-project는 명시적으로 설계에서 제외 (보안 이유).

3. **"Just-in-time context" 패러다임**: pre-embedded RAG에서 agent가 직접 grep/read하는 방향으로 전환 (Anthropic 공식 가이드).

4. **ICSE 2025 commit classification**: 10-타입 CC 분류 문제가 공식 연구 과제로 등장. CodeLlama + diff가 GPT-4보다 우수 — LLM size보다 domain fine-tuning이 중요.

5. **Hybrid retrieval 표준화**: BM25 단독 사용은 점차 deprecated. production system은 BM25 + dense + cross-encoder reranker 3단계 파이프라인.
